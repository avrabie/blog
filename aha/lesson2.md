## The "Chat" (A.K.A. Streaming Comments with Extra Steps)

Today's biggest feature: a chat page. Now, I need to be honest with you, dear reader. The backend for this "chat" is just... **streaming comments**. Nothing more, nothing less.
These streaming chats, on the BE have the same more or less code, just another POJO and a new streaming endpoing `/sse/chat/stream"`
At the moment I only have one room, but just like with the posts, we can create multiple rooms: same potato, different name.
And if you want to know how to do Streaming Comments: [Streaming Comment](https://blog.s4v3.net/posts/streaming-comments-with-spring-webflux-sinks-and-sse) which I know you did not read.

The frontend? Generated by [Opencode](https://opencode.ai). 

### How to Generate a Chat Page (A Prompt for the Ages)

If you want to create a chat page for your own blog using Opencode or Junie, here's the magic prompt:

```markdown
Create a chat page for my blog frontend that:
1. Connects to a Server-Sent Events (SSE) endpoint at `.GET("/sse/chat/stream", chatHandler::streamChats)`
2. Displays new comments in real-time as they arrive
3. Has a nice glass-morphism design that matches the rest of the blog
4. Uses React Query for managing the SSE connection
5. Shows a loading skeleton while connecting
6. Handles connection errors gracefully with a retry button
7. Displays the comment author's name, avatar (or initials), and timestamp
8. Auto-scrolls to new comments
```

And just like that, you have a chat page. Well, technically a "streaming comments with a slightly different UI" page, but who's counting?

## The Great Logout Mystery (To Be Continued...)

Here's a fun one for you. We implemented logout today. The button works. The UI changes. Everything seems fine.

And then you click login again.

And you're already logged in. (Feature not a Bug!)

*Mind blown*

So what gives? The session cookie, apparently. Even though logout "works", something in the gateway is holding onto your session cookie like a clingy ex. We'll need to investigate and properly invalidate that cookie on logout.

This is a bug for another day. Consider it a cliffhanger.
